# 스마트 예약 운영 데이터·구현 로드맵

작성일: 2026-07-20

상태: 검토용 기술 설계

연계 문서: [고도화 기능 기획서](2026-07-20-smart-reservation-feature-brief.md), [대시보드 시뮬레이션](2026-07-20-dashboard-simulation-scenarios.md)

## 1. 설계 원칙

- 기존 예약 검증이 최종 판단을 맡는다. 통계·재예약·추천은 검증을 우회하지 않는다.
- 집계는 서버에서 계산하고, 학생 화면에는 개인과 직접 관련된 결과만 전달한다.
- 1차 통계는 최근 4주 기준의 설명 가능한 규칙으로 계산한다. 머신러닝이나 개인 행동 예측은 사용하지 않는다.
- 통계 집계 실패가 예약·승인·반납 같은 기존 운영 흐름을 막아서는 안 된다.
- 추천과 알림은 작게 시작하고, 실제 사용 데이터로 기준을 조정한다.

## 2. 재사용할 현재 데이터

| 데이터 | 사용처 | 핵심 필드 |
|---|---|---|
| 예약 | 혼잡도·취소율·재예약·추천 | `type`, `status`, `reservedDate`, 시간 필드, `equipmentItemIds`, `createdAt` |
| 예약 timing | 반납 위험과 알림 | `startAt`, `endAt` |
| 기자재 | 가동률·대체 장비·부족 위험 | `id`, `category`, `active`, `reservable`, `status` |
| 차단 일정과 운영 설정 | 추천 가능 여부 | 시설별 시간대, 차단 일정, 대여 정책 |
| 사용자 | 즐겨찾기 소유자 식별 | 현재 로그인 사용자 ID |

새로 수집할 개인정보는 없다. 분석 응답에는 학생 이름, 학번, 전화번호를 포함하지 않는다.

## 3. 저장과 API 경계

### 3.1 최소 저장 필드

`user.preferences.favoriteEquipmentGroups`에 즐겨찾기 그룹 배열을 저장한다.

```json
[
  { "id": "favgrp_camera", "name": "카메라", "equipmentItemIds": ["eq_1", "eq_2"] },
  { "id": "favgrp_lens", "name": "렌즈", "equipmentItemIds": ["eq_3"] }
]
```

그룹은 최대 3개, 그룹명은 12자 이하, 그룹당 장비는 최대 5개다. 하나의 장비 ID는 한 그룹에만 들어갈 수 있다. 비활성 장비 ID는 홈에서 상태와 함께 표시하되 자동 삭제하지 않는다.

최근 예약과 동일 구성 재예약은 별도 테이블을 만들지 않고, 현재 사용자의 기존 예약에서 계산한다. 기존 예약을 참조하는 ID나 과거 개인정보를 새로 복제하지 않는다.

통계는 우선 요청 시 계산한다. 데이터 규모가 커져 응답 시간이 문제가 될 때만 일 단위 집계 캐시를 도입한다.

### 3.2 추가 API 제안

| API | 응답 또는 동작 | 권한 |
|---|---|---|
| `GET /api/admin/insights?from=&to=` | 혼잡 시간, 가동률, 취소율, 운영 경고 | 관리자 |
| `GET /api/me/reservation-shortcuts` | 최근 예약 3건, 즐겨찾기 그룹 최대 3개 | 승인 학생 |
| `PUT /api/me/favorite-equipment-groups` | 그룹 생성·이름 변경·장비 추가/이동/삭제가 반영된 전체 그룹 설정 저장 | 승인 학생 |
| `POST /api/reservations/recommendations` | 불가한 예약 초안에 대한 최대 3개 대안 | 승인 학생 |

추천 API는 실제 예약 생성 API와 분리한다. 학생이 대안을 선택한 뒤에도 기존 예약 생성 API가 다시 전체 검증을 수행한다.

## 4. 지표와 경고 계산

기본 분석 범위는 실행일을 포함한 최근 28일이다. 취소·반려 예약은 점유·가동률에서는 제외하고, 취소율에는 포함한다.

| 지표 | 계산 | 표시 조건 |
|---|---|---|
| 시간대 점유율 | 해당 시간대의 유효 예약 수 ÷ 해당 시간대의 예약 가능 수 | 예약 가능 수가 3 이상일 때 |
| 장비 가동률 | 장비가 유효 예약에 포함된 예약 일수 ÷ 해당 기간의 예약 가능 일수 | 활성·예약 가능 장비만 |
| 취소율 | `cancelled`, `admin_cancelled`, `rejected` 건수 ÷ 전체 신청 건수 | 신청이 5건 이상일 때 |
| 수요 증가 | 최근 7일 신청 수가 앞선 21일의 주간 환산 평균보다 30% 이상 증가하고 증가 건수가 3건 이상 | 장비 또는 카테고리별 |
| 부족 위험 | 가동률 80% 이상이며 같은 카테고리의 예약 가능 대체 장비가 1개 이하 | 상위 3개만 표시 |
| 미반납 위험 | 기자재가 `checked_out`이고 `endAt` 또는 반납 예정 시간이 현재보다 과거 | 즉시 표시 |

분모가 작거나 데이터가 모자랄 때는 0%를 꾸며서 보여 주지 않고 `데이터 부족`으로 표시한다.

## 5. 추천 알고리즘

추천 입력은 학생이 현재 보고 있는 예약 초안이다. 추천 전에 날짜 형식·예약 유형·장비 ID 같은 기본 입력만 확인한다.

```text
1. 현재 선택으로 실제 예약 검증을 실행한다.
2. 검증이 통과하면 추천을 만들지 않는다.
3. 동일 장비의 가까운 가능 시간·날짜를 최대 7일 범위에서 찾는다.
4. 없으면 같은 카테고리의 활성·예약 가능 장비를 찾는다.
5. 동일 날짜의 다른 가능 시간을 찾는다.
6. 각 후보를 기존 예약 검증 함수로 다시 검사한다.
7. 날짜 거리, 시간 거리, 동일 장비 여부 순으로 정렬해 최대 3개를 반환한다.
```

추천은 실제 다른 예약의 예약자 정보, 예약 사유, 연락처를 반환하지 않는다. `해당 시간은 이미 사용 중입니다`보다 `내일 13:00–15:00 가능`처럼 학생이 바로 행동할 수 있는 정보만 제공한다.

## 6. 알림 조정

현재 앱은 예약 데이터에서 네이티브 알림을 계획하고, 로그인·예약 변경·앱 동기화 때 갱신할 수 있다. 이 기반을 유지한다.

| 유형 | 예약 시점 | 제한 |
|---|---|---|
| 전날 알림 | 시작 24시간 전 | 예약당 최대 1개 |
| 사용 직전 | 시작 1시간 전 | 전날 알림과 12시간 이내면 생략 |
| 반납 임박 | 기자재 반납 1시간 전 | `checked_out`일 때만 유지 |
| 스튜디오 보고서 | 기존 사용 후·마감 임박 규칙 | 예약 알림 제한과 별개 |

예약 상태가 취소·반려·반납 완료로 바뀌면 해당 예약 ID로 잡힌 미래 알림을 모두 취소한다. 알림 권한이 없어도 예약 데이터와 관리자 통계는 정상 동작한다.

## 7. 단계별 구현 순서

### 단계 A: 집계 기반과 관리자 인사이트

1. 관리자 요약 API에 집계 함수와 `insights` 응답을 추가한다.
2. 단위 테스트로 상태별 포함·제외와 적은 표본 처리 기준을 고정한다.
3. 현재 `AdminDashboard`에 인사이트와 주의 필요 카드, 기존 목록으로의 필터 이동을 추가한다.

### 단계 B: 학생 재예약과 그룹형 즐겨찾기

1. 사용자 환경설정에 즐겨찾기 그룹 배열을 추가하고, 그룹 3개·그룹당 장비 5개·장비 중복 금지 검증을 넣는다.
2. 바텀시트에서 그룹 선택·생성·이름 변경·장비 검색·이동을 처리한다.
3. 최근 예약을 안전한 초안으로 변환하는 함수를 만든다.
4. 학생 홈에 `다시 예약` 영역을 추가하고, 현재 예약 폼의 검증과 연결한다.

### 단계 C: 알림 빈도 제한

1. 현재 알림 계획기의 예약 전 알림을 전날·1시간 전·10분 전·시작에서 정책에 맞게 줄인다.
2. 12시간 병합과 상태 변경 시 취소를 테스트한다.
3. 학생·관리자 설정에서 예정 알림 수가 바르게 갱신되는지 확인한다.

### 단계 D: 결정형 추천

1. 기존 예약 검증 로직을 호출하는 추천 후보 생성 함수를 구현한다.
2. 예약 실패 응답 또는 별도 추천 API에서 최대 3개를 반환한다.
3. 학생 예약 화면과 재예약 흐름에 추천 카드를 추가한다.

각 단계는 독립 배포 가능해야 한다. D 단계는 A 단계의 통계를 읽을 수 있지만, 추천의 최종 가능 여부를 통계에 의존하지 않는다.

## 8. 테스트와 수용 기준

| 영역 | 확인 내용 |
|---|---|
| 도메인 | 취소 예약이 가동률에는 빠지고 취소율에는 포함되는지, 수리중 장비가 대체 후보에서 제외되는지 |
| 권한 | 학생이 관리자 인사이트에 접근할 수 없고, 다른 학생의 즐겨찾기 그룹을 수정할 수 없는지 |
| 즐겨찾기 그룹 | 그룹 3개·그룹당 장비 5개 한도, 장비의 그룹 간 중복 금지, 이동과 바텀시트 포커스 복귀가 동작하는지 |
| 추천 | 추천된 모든 후보가 같은 검증 함수에서 통과하고, 유효 후보가 없을 때 빈 목록을 정직하게 반환하는지 |
| 알림 | 12시간 이내 중복 제거, 취소·반납 완료 시 미래 알림 제거, 보고서 알림 유지 |
| 화면 | 모바일 390px·430px와 데스크톱에서 카드·표·터치 영역·빈 상태가 읽히는지 |
| 회귀 | 기존 `npm run test:backend-domains`, `npm run test:admin-ui`, `npm run test:student-react`, `npm run test:notifications`를 통과하는지 |

## 9. 운영상 주의점

- 대시보드 지표는 운영 판단을 돕는 신호이지 자동 취소·자동 벌점의 근거가 아니다.
- 장비별 데이터가 적은 초기에 순위와 추세를 과신하지 않는다.
- 고가 장비의 대체 후보는 같은 카테고리라는 이유만으로 추천하지 않는다. 현재 장비 정책과 예약 가능 상태를 먼저 통과해야 한다.
- 집계 응답이 실패하면 기존 대시보드의 승인·대여·반납 처리 큐는 정상적으로 계속 보여야 한다.
